What is Design Patterns

  • Design patterns are used to represent some of the best practices adapted by experienced object-oriented software developers. A design pattern systematically names, motivates, and explains a general design that addresses a recurring design problem in object-oriented systems. It describes the problem, the solution, when to apply the solution, and its consequences. It also gives implementation hints and examples.

Factory Pattern (in calculator)

Python file reference : operation.py

  • Class: Operation
  • Factory method is a creational design pattern, i.e., related to object creation. In Factory pattern, we create objects without exposing the creation logic to the client and the client uses the same common interface to create a new type of object.
  • The idea is to use a static member-function (static factory method) that creates & returns instances, hiding the details of class modules from the user.
  • A factory pattern is one of the core design principles to create an object, allowing clients to create objects of a library(explained below) in a way such that it doesnâ€™t have tight coupling with the class hierarchy of the library.

Example

  • Class Operation has the factory method for creating the instance of Operation, Addition, Subtraction, Multiplication and Division etc.
  • class Operation is the base/ parent class
  • @classmethod
  • def create_operation(cls, values: tuple):
  •     """design pattern - factory to create the instance at one place"""
  •     # cls keyword (ideal name) : we can only access the members of the class
  •     return cls(values)
  • the factory method is called in the calculations.py (Calculations class) whenever the operation(Add, Subtract etc ) added to the static history variable.
  • add_subtraction_calculation of class Calculations is calling the factory method
    of Operation class
  • @staticmethod
  • def add_subtraction_calculation(tuple_elements):
  •     """Add subtraction object to history"""
  •     subtract = Subtraction.create_operation(tuple_elements)
  •     Calculations.add_calculation(subtract)
  •     return True

Advantages:

  • factory to create the instance at one place instead of scattering all over the code.

Facade (in calculator)

Python file reference : calculator.py

  • Software entities should be open for extension but closed for modification.
  • You should not need to modify the code you have already written to accommodate new functionality,but simply add what you now need.
  • This does not mean that you cannot change your code when the code premises needs to be modified, but that if you need to add new functions similar to the one present, you should not require to change other parts of the code.

Example

  • class Calculator:

  •     def add_numbers(tuple_elements: tuple):....
  •     def subtract_numbers(tuple_elements: tuple):...
  •     def multiply_numbers(tuple_elements: tuple):...
  •     def divide_numbers(tuple_elements: tuple):...

  • If we want to add new functionality such as SquareRoot and Mean or Square or Median etc in the Calculator class, then we are just needed to add the new method instead of changing the existing method add, subtract, multiply and divide.None of the existing function need to be modified when we need to add new functionality.No existing functions in the other classes such as Calculations (for history) needs to be modified. that is why we call that Software entities should be open for extension but closed for modification.

Advantage

  • The result is a very flexible class, that requires minimum time to be maintained.

Command Patern (in calculator)

Python file reference : Operation.py and Addition.py

  • Functions that use pointers or references to base classes must be able to use objects of derived classes without knowing it.
  • Derived classes must be substitutable for their base classes
  • if a subclass redefines a function also present in the parent class, a client-user should not be noticing any difference in behaviour, and it is a substitute for the base class.For example, if you are using a function and your colleague change the base class, you should not notice any difference in the function that you are using.

Example (Pls note that this is not in Calculator but just to explain the principle

  • class Operator:
  •     # Base Class
  •     def __init__(self, values: tuple):
  •         self._values = Operation.convert_cal_input_to_float(values)

  • class Addition:
  •     # Subclass Class
  •     def __init__(self, values: tuple, name: String):
  •         super().__init__(self, values)
  •         self.name = name

  • If we are using a = Operation((1.0, 2.0, 3.0))
  • then we can use a = Addition((1.0, 2.0, 3.0), "name")

Advantage

  • The result of this principle is that weâ€™d write our code in a consistent manner and, the end-user will need to learn how our code works

The Interface Segregation Principle (ISP) (in calculator)

Python file reference : result.py and addition.py

  • Clients should not be forced to depend upon interfaces that they do not use.

Example

  • class Result:
  •     """result for all the operations"""

  •     @abstractmethod
  •     def get_output(self):
  •         """this is the abstract method to be implemented by impl classes"""

  • class Addition:
  •     """subclass Addition extending base Operation"""

  •     # This is the instance method
  •     def get_output(self):
  •         """Adds the elements in tuple and return"""
  •         sum_of_elements = 0.0
  •         for value in self._values:
  •         sum_of_elements = sum_of_elements + value
  •         return sum_of_elements

  • Result is the abstract class having abstract method get_output().This has specific method that is needed by all operation of the subclass - Addition etc.Classes are not forced to implement any methods from the interface that are not supposed to be part of their behaviour.

Advantage

  • Final aim to keep our classes clean and minimise mistakes.

Dependency inversion principle (in calculator)

Python file reference : calculator_history-test.py

  • Abstractions should not depend on details. Details should depend on abstraction. High-level modules should not depend on low-level modules. Both should depend on abstractions
  • Dependency Injection (DI) is a programming technique that makes a class independent of its dependencies. In software engineering, dependency injection is a technique whereby one object supplies the dependencies of another object. A â€˜dependencyâ€™ is an object that can be used, for example as a service. Instead of a client specifying which service it will use, something tells the client what service to use. The â€˜injectionâ€™ refers to the passing of a dependency (a service) into the object (a client) that would use it
  • High coupling. If the coupling is high itâ€™s like using a superglue or welding. No easy way to disassemble.
  • High cohesion. High cohesion is like using the screws. Very easy to disassemble and assemble back or assemble a different way. It is an opposite to high coupling.
  • good reference at https://python-dependency-injector.ets-labs.org/introduction/di_in_python.html
